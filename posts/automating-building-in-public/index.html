Automating Building in Public

August 14, 2021 3-minute read
Hugo • Github • Python

Introduction

As I mentioned in my previous post, I have begun building my personal site to post projects I’m working on. I’ve recently been learning about “Building in Public” and central to it is to maintain constant communication with the public. I thought that it would be great if I could post links to my personal site on Twitter. That way my followers could instantly see any new updates I’ve made.

However, the process seemed somewhat tedious and also error-prone. There would be no way to keep myself accountable to keep both the site and Twitter in sync. So what if there was a way to keep them in sync? How about everytime I published a new post, a bot could post a link to my Twitter feed? Sounded simple enough.

A few hours later and I’ve built a rudimentary integration between Github and Twitter. Everytime I commit a new post, I take a diff of the links on my site. Any new ones, I post to Twitter. Problem solved!

How it works

Summary

The main components to the solution are:

  • Github Actions that run after a commit is pushed
  • Using git history to find the most recent and second most recent index.xml RSS file generated by Hugo.
  • Running a Python script that takes the diff between those two files.
  • Using tweepy and API tokens and secrets to interact with the Twitter API.

The greatest thing about this solution is that it doesn’t cost me anything!

Github Actions

I already utilize Github Actions to automatically build and publish my private site source code to my public Github pages repository. After that, I added the following steps to first checkout the public repository. I’m using a depth of 2 to later get the older index file.

name: Checkout for RSS
uses: actions/checkout@v2
with:
    repository: hworden/hworden.github.io
    fetch-depth: 2
    path: hugo_site

I then ensure that python is installed on the Github action worker by including the next step. 

name: Setup Python
uses: actions/setup-python@v2
with:
    python-version: '2.7'

Lastly, I run this fairly complicated script in the run step. 

name: Echo
run: |
    cd hugo_site
    cp index.xml ../website-poster/index1.xml
    git checkout gh-pages~1
    cp index.xml ../website-poster/index2.xml
    cd ../website-poster
    pip install -r requirements.txt
    python tweeter.py --v --send-tweets --consumer-key ${{ secrets.TWITTER_CONSUMER_KEY }} --consumer-secret-key ${{ secrets.TWITTER_CONSUMER_SECRET_KEY }} --access-token ${{ secrets.TWITTER_ACCESS_TOKEN }} --access-token-secret ${{ secrets.TWITTER_ACCESS_TOKEN_SECRET }} index2.xml index1.xml

This last one is complicated… I basically use git to get (no pun intended) the index.xml file from the HEAD of that repo and the commit preceding it. I then just copy them somewhere where I can get access to them.

Next I navigate to my website-poster directory. This is a submodule that I’ve included in my git repo that contains my tweeter.py script.

The script takes:

  • Twitter App Consumer Key + Twitter App Consumer Secret Key
  • Twitter Access Token + Twitter Access Token Secret
  • the two index.xml files

The secrets are all managed by Github’s Secret Management and are not hardcoded into the script. Internally, the script diffs them and then uses tweepy to make a status update. The script itself could be the topic of it’s own post… (let me know on Twitter if you’d like more info!).
